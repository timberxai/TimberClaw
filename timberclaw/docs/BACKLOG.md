## M0 · 基础设施

### M0-01 Builder 项目脚手架（OpenHands fork + TimberClaw 业务模块）
- **范围**：基于 **OpenHands** fork（锁定具体 release tag）；叠加 Django + PostgreSQL 业务后端；前端在 OpenHands 原 React 工程基础上引入 **Ant Design v5**，为 TimberClaw 新增业务屏建立独立路由前缀（例如 `/tc/*`）与独立布局壳；统一 `docker-compose.yml` 可同时起 OpenHands 服务、Django 业务后端、PG、前端
- **不做**：CI/CD、自动发布、多环境推送、改写 OpenHands 原有 UI 风格
- **依赖**：无
- **对齐 PRD**：§2、§6.1、§6.1.1、§9.4
- **验收**：`docker compose up` 可启动 Builder 本体（OpenHands + Django + PG + 前端）；能访问 OpenHands 原有屏与 `/tc/*` 下的 AntD 占位页

### M0-02 Builder 认证与账号
- **范围**：Django 内置 auth + 账密登录；五类角色（Owner/Reviewer/Admin/Platform Engineer/Human Developer）的角色模型；角色 ↔ 动作级权限
- **不做**：SSO / OAuth / LDAP、自定义角色
- **依赖**：M0-01
- **对齐 PRD**：§4、§6.2
- **验收**：五类角色均可登录；不同角色访问受限资源被正确拒绝

### M0-03 LLM Gateway 抽象
- **范围**：统一 LLM 调用出口；至少支持两家厂商（默认 OpenAI + 一家国内厂商）；Token 用量 / 成功率 / 失败原因监控；单任务最大 Token / 最大时长上限；出站前做敏感字段过滤（凭据、内网主机名、业务真实数据样本）
- **不做**：多模型路由策略、Prompt 缓存、向量检索
- **依赖**：M0-01
- **对齐 PRD**：§8.5 "LLM 与执行底座"
- **验收**：切换 LLM 厂商只改配置；Builder 后台可查看调用统计；设置极小 Token 上限能触发终止

### M0-04 GitLab 接入
- **范围**：配置 GitLab URL + Project Access Token；连通性自检；以机器账号身份创建分支、commit、MR
- **不做**：GitLab Webhook 回调、合并事件订阅（M3 再做）
- **依赖**：M0-01
- **对齐 PRD**：§8.7
- **验收**：Builder 可在指定测试仓库创建分支、推 commit、开 MR，且身份为机器账号

### M0-05 Builder 部署与自检
- **范围**：`docker-compose.yml` + `README` 部署说明；启动自检脚本（PG 连接、GitLab 连通、LLM 连通）
- **不做**：生产级高可用、异地备份
- **依赖**：M0-01、M0-03、M0-04
- **对齐 PRD**：§9.4
- **验收**：新机器上 30 分钟内完成部署并通过自检

## M1 · Spec 管道

### M1-01 需求输入中心
- **范围**：上传设计文档（pdf/docx/md）、文本输入、对话输入、选择基础业务模板；原始需求 + 需求摘要 + 待确认问题的持久化
- **不做**：OCR、复杂图文解析、设计稿可视化理解
- **依赖**：M0-02、M0-03
- **对齐 PRD**：§8.1
- **验收**：任一输入方式均可生成原始需求包 + 摘要 + 待确认问题列表

### M1-02 Spec Analyst 与初始 spec 生成
- **范围**：从需求包生成符合 §8.2 最小内容的初始 spec（应用信息 / 实体字段 / 页面 / 角色权限 / 流程状态机 / 校验 / 报表 / 集成 / 环境）
- **不做**：跨项目 spec 复用、spec 自动合并
- **依赖**：M1-01
- **对齐 PRD**：§8.2、§8.5（Spec Analyst）
- **验收**：对"工单 + 报工 + 质检"示例需求，能生成结构完整且可读的初始 spec

### M1-03 对话修订 spec
- **范围**：自然语言对话修订 spec；每次修订产生新版本与 diff；对话记录留存
- **不做**：多人协同对话、对话分叉
- **依赖**：M1-02
- **对齐 PRD**：§5.1、§8.2
- **验收**：用户可通过一句话要求新增字段 / 调整状态，spec 正确更新并展示 diff

### M1-04 结构化 spec 编辑
- **范围**：表格化的结构编辑器（实体/字段/页面/权限/状态机）；前端 React 18 + TypeScript + Ant Design v5 实现（优先复用 ProTable / ProForm）
- **不做**：可视化流程画布
- **依赖**：M1-02
- **对齐 PRD**：§5.1、§8.2
- **验收**：可在编辑器内完成与对话修订等价的 spec 改动

### M1-05 Spec 版本化与确认闸门
- **范围**：spec 版本表 + diff 视图；Owner 的"确认当前 spec"动作；未确认版本不进入 M2
- **不做**：版本回滚、版本分支
- **依赖**：M1-03、M1-04
- **对齐 PRD**：§5.1、§6.3 第 6 条、§8.2
- **验收**：未确认的 spec 无法触发代码生成；确认后状态不可逆（除非走修订流程）

### M1-06 Spec 双视图（V1.5 新增）
- **范围**：
  - Spec 底层模型保持**一份**（结构化 JSON / 表），不新增双存储
  - Builder 前端实现两套视图：
    1. **业务场景视图**（Owner 默认入口）：以"业务场景卡片"呈现，一个场景 = {角色 / 触发 / 主路径 / 异常路径 / 衡量指标}；支持对话式修订
    2. **结构化专业视图**（Platform Engineer / Owner 次级入口）：展开实体、字段、页面清单、状态机、权限矩阵、报表 / 仪表盘定义、主数据来源声明；支持表单编辑
  - 业务场景视图提供"涉及的数据与页面"只读展开面板，让 Owner 在不切视图的情况下预览专业视图摘要
  - spec diff 必须输出**两份摘要**：业务场景视图摘要（供 Owner / Reviewer）+ 结构化专业视图 diff（供 PE）
  - 从任一视图的改动必须能反推出另一视图的对应摘要；反推失败时由 Agent 主动要求 Owner 澄清
- **不做**：
  - spec 双存储或双仓
  - 业务场景视图的可视化流程画布
  - Reviewer 级的视图访问入口（Reviewer 不看 spec，只看候选版本人话说明）
- **依赖**：M1-02、M1-03、M1-04
- **对齐 PRD**：§8.2（双视图）、§5.8（概念隔离）、§1.2 术语表
- **验收**：
  - Owner 以业务场景视图为默认；一句话修订能触达业务场景卡片与结构化专业视图；diff 摘要双份生成
  - 结构化专业视图中添加字段后，业务场景视图自动得到"涉及数据已更新"的一句话摘要
  - Reviewer 账号下无法进入任一 spec 视图

## M2 · 模板底座 + 首次代码生成

### M2-01 前端模板底座
- **范围**：React 18 + TypeScript + Ant Design v5 脚手架（与 Builder 前端同栈）；列表（ProTable）/ 表单（ProForm）/ 详情 / 简单报表四种通用页面模板；统一导航（ProLayout）、状态标签、操作区组件；工业风主题（§5.7）
- **不做**：自定义主题编辑器、移动端布局
- **依赖**：M0-01
- **对齐 PRD**：§5.7、§6.1、§8.3
- **验收**：模板工程独立可跑；四种页面用 mock 数据渲染符合工业风规范

### M2-02 后端模板底座
- **范围**：Django 脚手架；Auth / RBAC / 实体 CRUD / 状态流转 / 操作日志；PostgreSQL 连接；Django Migrations；pytest-django 测试基线
- **不做**：Celery、消息队列、缓存层
- **依赖**：M0-01
- **对齐 PRD**：§6.1、§8.3
- **验收**：脚手架项目可运行、API 文档自动生成、可跑通基线测试

### M2-03 部署适配层模板
- **范围**：Preview / Prod 共用的 `docker-compose.yml` 模板（应用 + PostgreSQL + 后台任务占位）；环境变量约定；启停脚本
- **不做**：K8s、Helm、Serverless
- **依赖**：M2-01、M2-02
- **对齐 PRD**：§6.1、§8.3、§8.8、§8.11
- **验收**：用模板填参后可在本地一条命令起一个完整应用

### M2-04 App Architect
- **范围**：从 spec 生成"实现计划"（要新增哪些 Model / API / 页面 / 路由）；计划可 Owner 查看
- **不做**：多方案对比、成本估算
- **依赖**：M1-05、M2-01、M2-02
- **对齐 PRD**：§8.5（App Architect）
- **验收**：对工单 spec 生成的计划清晰、可执行、不含未被需求使用的项

### M2-05 Code Builder 首次生成
- **范围**：按实现计划在模板底座内生成实体 / migration / API / 页面；禁止在模板外创建新子系统
- **不做**：自定义架构、新中间件引入
- **依赖**：M2-04
- **对齐 PRD**：§5.4、§5.6、§8.4、§11
- **验收**：对工单 spec 生成的代码可本地起服务、基础 CRUD 可用

### M2-06 fixtures 生成器
- **范围**：按 spec 生成 3–5 条覆盖主要状态的样例数据；Preview 部署时加载
- **不做**：从生产数据脱敏、大规模数据
- **依赖**：M2-05
- **对齐 PRD**：§8.8（样例数据策略）
- **验收**：每个核心对象至少出现 1 条处于每种主要状态的 fixture

### M2-07 仪表盘一等公民模板（V1.5 新增）
- **范围**：
  - 在前端模板底座（M2-01）中新增"仪表盘页面"为与列表 / 表单 / 详情并列的一等公民
  - 仅引入 **Ant Design Charts（或 AntV G2Plot）** 一家图表库；**不引入** ECharts
  - 支持 MVP 图表类型：柱图、折线图、饼图 / 环形图、KPI 数字卡（数字 + 环比）、明细表
  - spec 的结构化专业视图内新增 `dashboards[]` 段，每项声明：数据源（实体 / 聚合 API）、指标（计数 / 求和 / 均值 / 合格率 / 时间段增长率）、维度（日 / 班次 / 工作中心 / 物料 / 工单状态）、过滤、图表类型
  - 所有聚合在后端完成，提供通用 `GET /api/dashboards/<id>` 聚合接口；前端仅做渲染
  - 支持"点击图表一项下钻到对应列表页"，URL 自动带维度过滤参数
  - 在 App Architect 与 Code Builder（M2-04 / M2-05）中增加对 `dashboards[]` 的生成支持
  - 性能 SLA：Prod 数据量 ≤ 100 万行时单面板 P95 ≤ 3 秒（未达标由 Platform Engineer 加索引 / 预聚合处理，不换框架）
- **不做**：拖拽式仪表盘设计器、业务使用者自定义指标、跨项目看板共享、实时流式更新
- **依赖**：M2-01（前端模板底座）、M2-02（后端模板底座）、M2-04 / M2-05（生成链路）
- **对齐 PRD**：§8.3.1（仪表盘一等公民）、§12.1、§12.3
- **验收**：
  - 示例 spec 声明 3 个仪表盘（当日产量、工序合格率、返工率），能生成出可在 Preview 跑通的页面
  - 4 类图表 + KPI 卡 + 明细均覆盖
  - 下钻跳转到列表页并保留维度过滤
  - 100 万行数据下 P95 ≤ 3 秒（用 fixtures 放大版验证）

### M2-08 主数据 CSV / Excel 导入与结果 CSV 导出（V1.5 新增）
- **范围**：
  - **实体声明来源**：spec 的每个实体加 `source ∈ { internal | external_import | mixed }`；Code Builder 据此决定 CRUD 能力
  - **导入能力**：
    - 模板底座提供通用"主数据导入页"组件；每个 `external_import` / `mixed` 实体自动挂载
    - 可下载**导入模板**（列名 + 必填标记 + 示例值）
    - 上传后做**预校验**（列名匹配、必填校验、枚举值校验、业务键唯一性）；不校验通过的单行不落盘
    - **业务键 upsert** 语义保证幂等；支持失败行 CSV 下载（含错误原因列）
    - 操作日志记录：谁 / 何时 / 成功行数 / 失败行数 / 来源文件名
    - MVP 仅支持人工触发上传（不做定时 / 接口导入）
  - **导出能力**：
    - 所有列表页默认具备"按当前筛选条件导出 CSV"
    - 仪表盘页面支持导出"明细 + 聚合"CSV
    - 导出不包含内部主键 / SHA；业务键必须在首列
    - 文件名规则：`<entity>_<yyyymmdd>_<hhmm>.csv`，UTF-8 + BOM
  - **页面约束**：`external_import` 实体的 CRUD 页**隐藏**"新增 / 批量删除"按钮，仅保留查看 / 筛选 / 导出 / 导入
- **不做**：ERP 接口对接、主数据变更回推外部系统、自定义列映射器、复杂主数据治理 / 主数据版本管理
- **依赖**：M2-01、M2-02、M2-05
- **对齐 PRD**：§8.12、§3.3（离散制造主数据：物料 / 工作中心 / 员工 / 班组）、§12.1、§12.3
- **验收**：
  - 离散制造首期 4 类主数据（物料 / 工作中心 / 员工 / 班组）可通过 CSV 导入，失败行能下载并重录
  - 同业务键二次导入走 upsert，不重复创建
  - `external_import` 实体的 CRUD 页无新增按钮
  - 工单列表在带筛选条件下导出 CSV，UTF-8 + BOM，用中文 Excel 打开不乱码

## M3 · Dev 验证 + Git/PR

### M3-01 Dev 验证编排
- **范围**：串联 lint（ruff + eslint）、typecheck（mypy + `tsc --noEmit`）、build（vite build + `manage.py check`）、migration 检查、pytest、vitest
- **不做**：端到端 UI 测试、压测
- **依赖**：M2-05
- **对齐 PRD**：§8.6
- **验收**：任一步失败都能结构化上报，通过率记录进监控

### M3-02 Repair Agent 自动修复
- **范围**：对确定性错误（lint / 类型 / 显式异常）自动修复，最多 2 次重试；超过即转人工
- **不做**：架构级重构修复
- **依赖**：M3-01
- **对齐 PRD**：§8.5（Repair Agent）、§8.6
- **验收**：样例故意制造的 lint / 类型错误能被自动修好

### M3-03 代码层面高风险识别
- **范围**：按 §8.6 清单识别高风险改动（高风险 migration、认证授权加密审计目录改动、新增一级依赖、模板底座改动、连续两次修复失败）；命中即终止并通知 Owner
- **不做**：动态策略学习
- **依赖**：M3-01
- **对齐 PRD**：§8.6 "代码层面高风险"
- **验收**：构造每类高风险场景均能被识别并阻断

### M3-04 分支与 MR 自动化
- **范围**：分支命名 `tc/<spec_version>/<short_desc>`；自动 commit + 首次 MR 创建；MR 描述含 spec 版本号；patch 持续更新同一 MR
- **不做**：多 MR 合并、cherry-pick
- **依赖**：M0-04、M2-05
- **对齐 PRD**：§8.7
- **验收**：生成-提交-MR 一条龙完成，MR 上能看到 spec 版本链接

### M3-05 Dev 失败通知分派（V1.5 新增）
- **范围**：
  - Dev 验证失败 / 高风险识别命中 / 连续 2 次自动修复失败时，按如下规则分派：
    1. 首要接盘：**Platform Engineer**（工程错误、依赖冲突、migration 报错、模板底座异常、LLM Gateway 故障）
    2. 次要接盘：**Admin**（账号 / 环境配置 / 权限配置类问题）
    3. Owner 被"通知"但不是接盘人；唯一要求 Owner 决策的场景是：Agent 判定"继续修"需要修改 spec 范围
  - 通知内容按角色分发：
    - 发给 Platform Engineer：原始工程术语（堆栈、diff、lint 报错、失败日志链接）
    - 发给 Owner：按 §7.4 映射后的人话（例如"系统自检发现问题，工程师已介入处理"）
  - 通知渠道 MVP 支持：站内消息 + 邮件；**不做** 飞书 / 钉钉 / 企微集成
  - 失败单有"接盘状态"字段：未接盘 / 处理中 / 已关闭 / 已升级到 Owner 决策
  - Owner 界面**不显示**未分派给自己的失败单细节；Platform Engineer / Admin 可看全局视图
- **不做**：外部 IM 集成、值班排班、SLA 告警升级
- **依赖**：M3-01、M3-02、M3-03
- **对齐 PRD**：§8.6 "转人工"、§5.8（概念隔离）
- **验收**：
  - 构造 migration 高风险场景，失败单进入 Platform Engineer 视图，Owner 只收到人话摘要
  - Owner 视图里点失败单只能看到人话；工程术语必须在 Platform Engineer 视图才出现
  - Owner 仅在"需改 spec 范围"场景下被拉进决策

## M4 · 候选版本 + Preview

### M4-01 候选版本数据模型
- **范围**：`ReleaseCandidate (project, pr_id, commit_sha, rc_number, status, spec_version)` 模型；rc_number 按 PR 内部递增；支持"废弃"状态
- **不做**：跨 PR 继承 rc 编号
- **依赖**：M3-04
- **对齐 PRD**：§7.1
- **验收**：同一 PR 内多次创建 rc 编号递增；废弃后不再参与部署

### M4-02 Owner 创建候选版本 UI
- **范围**：PR 详情页选择某 commit 创建 rc；前置校验 Dev 验证通过
- **不做**：Reviewer 触发 rc
- **依赖**：M4-01
- **对齐 PRD**：§7.2、§7.3
- **验收**：Dev 未通过时按钮禁用；创建成功后返回 rc-N

### M4-03 Preview Runner
- **范围**：本地 Preview Runner 节点抽象；通过 Docker Compose 启/停一组 Preview 服务；每个 Preview 独立 PG 实例或独立 schema；Preview URL + 状态 + 日志视图
- **不做**：Kubernetes、多节点调度
- **依赖**：M2-03、M4-02
- **对齐 PRD**：§8.8（部署形态）
- **验收**：同一项目可并存 ≥2 个 Preview，互不干扰

### M4-04 Preview 生命周期
- **范围**：默认 TTL 7 天 + 手动延长；rc 废弃时立即回收；部署失败自动重试 1 次后告警
- **不做**：按使用频率自动延长
- **依赖**：M4-03
- **对齐 PRD**：§8.8（生命周期）
- **验收**：到期 Preview 自动下线，废弃 rc 的 Preview 立即下线，失败重试有日志

### M4-05 fixtures 加载
- **范围**：Preview 首次部署自动加载 fixtures
- **不做**：随时重置
- **依赖**：M2-06、M4-03
- **对齐 PRD**：§8.8（样例数据策略）
- **验收**：Preview 页面能看到 fixtures 数据

### M4-06 Owner UI 语言映射层（V1.5 新增）
- **范围**：
  - 在 Builder 前端提供**全局术语映射层**（i18n 式键值 + 角色上下文），覆盖 §7.4 全部条目
  - 面向 Owner 的所有业务屏（需求输入、spec、候选版本、Preview、发布、反馈、通知、审计摘要）必须经过映射层渲染；禁止在组件代码里直接硬编码 `commit`、`PR`、`rc`、`migration` 等英文工程词
  - 面向 Reviewer 的 Preview 内说明（版本说明、反馈面板）走**更严版映射**：彻底屏蔽 rc 编号替换为"版本 N"
  - 面向 Platform Engineer / Admin 的专区不走映射（允许用原词）
  - 生成系统（Code Builder 产物）一律使用**最严版映射**：禁止任何工程术语泄漏；错误提示模板统一走底座提供的文案
  - 提供 ESLint / 文本扫描检查规则：检测 Owner / Reviewer / 生成系统目录下是否出现未经映射的敏感词清单（`commit`, `rc-`, `PR `, `merge request`, `migration`, `diff` 等），作为 Dev 验证的一步
- **不做**：多语言国际化、按 Owner 喜好自定义术语
- **依赖**：M2-01（前端模板底座）、M4-01 / M4-02（候选版本）
- **对齐 PRD**：§5.8（概念隔离）、§7.4（术语映射表）、§13.2（概念隔离达标率）
- **验收**：
  - Owner 屏中点击"开启在线试用"实际触发 Preview 部署；没有任何地方出现 `Preview` / `rc-N` 英文
  - Reviewer 进入 Preview 内置说明："当前版本 3，相比版本 2 调整了……"；无 commit / PR 字眼
  - 生成系统的报工页错误提示不会出现 `migration failed` / `stack trace` 之类文案
  - ESLint / 文本扫描规则能够在 Dev 验证中捕捉未经映射的工程词

## M5 · 反馈 → patch 闭环

### M5-01 Preview 反馈面板
- **范围**：Preview 每页注入反馈按钮；独立面板采集反馈类型 + 描述；自动带入 URL / 页面路径；可选组件标识
- **不做**：页面内自由标注
- **依赖**：M4-03
- **对齐 PRD**：§8.9
- **验收**：任意 Preview 页面均能提交反馈；数据含 §8.9 "必需数据"全部字段

### M5-02 反馈状态机
- **范围**：`待处理 → 处理中 → 已 patch / 已拒绝 / 已关闭`
- **不做**：反馈打标签、分组统计
- **依赖**：M5-01
- **对齐 PRD**：§8.9、§8.10
- **验收**：状态流转受角色约束，非法流转被拒绝

### M5-03 风险判定
- **范围**：静态规则（migration / 认证授权审计目录 / 状态机定义 / 模板底座文件命中 ⇒ 高风险）+ Agent 研判；Owner 可人工升降级
- **不做**：基于历史样本的机器学习判定
- **依赖**：M5-02
- **对齐 PRD**：§8.10 "风险判定"
- **验收**：命中静态规则时 Agent 无法降级；Owner 可人工升级

### M5-04 低风险自动 patch
- **范围**：文案 / 展示顺序 / 简单布局 / 筛选展示 / 轻量交互文案类 ⇒ Repair Agent 自动 patch ⇒ 更新 MR ⇒ 重跑 Dev 验证
- **不做**：跨页面联动改动
- **依赖**：M5-03、M3-01、M3-04
- **对齐 PRD**：§8.10（低风险修改）
- **验收**：构造文案反馈，能自动出 patch 并出现在 MR 上

### M5-05 高风险走 spec → 代码
- **范围**：高风险反馈强制先走 spec 修订再驱动代码
- **不做**：绕过 spec 直接改代码
- **依赖**：M1-03/04/05、M5-03
- **对齐 PRD**：§8.10（高风险修改）
- **验收**：改数据模型 / 状态机的反馈必须先看到 spec diff 并被 Owner 确认，才能出代码 patch

## M6 · Prod 发布 + 回滚

### M6-01 Prod 发布页
- **范围**：汇总 rc / PR / spec 版本 / Dev 验证 / 审计信息；备注必填；人工确认后发布
- **不做**：双人审批
- **依赖**：M4-01、M7-04
- **对齐 PRD**：§8.11、§6.3 第 11 条
- **验收**：未填备注或 rc 未验收时按钮禁用

### M6-02 Prod 部署执行
- **范围**：基于 M2-03 模板做 Prod Docker Compose 部署；Prod 数据库独立管理；与 Preview 完全隔离
- **不做**：蓝绿 / 金丝雀
- **依赖**：M2-03、M6-01
- **对齐 PRD**：§8.11、§9.3
- **验收**：首次发布成功；数据库连接走 Prod 独立配置

### M6-03 回滚
- **范围**：一步回滚到上一个在 Prod 生效过的 rc（不跳版）；回滚备注必填；审计来源 `rollback`
- **不做**：跳版回滚、任意版本回滚
- **依赖**：M6-02、M7-04
- **对齐 PRD**：§8.11（回滚）
- **验收**：常规回滚全自动；不可逆 migration 场景必须强制进入 Platform Engineer 确认分支

## M7 · 数据治理 + 审计

### M7-01 数据合同注册表
- **范围**：核心对象必须声明主键 / 业务键 / 字段清单 / 状态枚举；缺失则不允许进 Prod
- **不做**：owner system / 同步策略（§10.2 明确 MVP 不要求）
- **依赖**：M2-05
- **对齐 PRD**：§10.2
- **验收**：缺失数据合同的对象在 M6-01 被阻断

### M7-02 migration 高风险识别
- **范围**：按 §10.3 清单识别（删字段 / 改类型 / 改主键或唯一约束 / 拆表合表 / 改关键状态枚举语义）；命中必须人工确认
- **不做**：自动前向兼容重写
- **依赖**：M3-03
- **对齐 PRD**：§10.3
- **验收**：构造五类高风险 migration 均能被识别

### M7-03 写入边界
- **范围**：Prod 写入只允许走受控 API / 服务；状态机规则在服务层生效；权限控制到角色 × 动作
- **不做**：字段级权限、行级权限
- **依赖**：M2-02、M0-02
- **对齐 PRD**：§10.4
- **验收**：跳过服务层的写入尝试被拒绝

### M7-04 操作日志
- **范围**：记录 who / when / what / target / source；source 枚举：`human_ui` / `agent_patch` / `migration_job` / `rollback`
- **不做**：字段级前后值全量审计
- **依赖**：M0-02
- **对齐 PRD**：§10.5、§8.11（回滚审计）
- **验收**：每个关键动作均落日志；日志可按 source 过滤

### M7-05 保留与备份
- **范围**：操作日志保留 ≥180 天；Prod PG 每日全量备份、保留 ≥30 天；备份存独立路径
- **不做**：异地 / 多副本 / point-in-time
- **依赖**：M6-02
- **对齐 PRD**：§9.5、§10.6
- **验收**：备份恢复演练成功一次

### M7-06 Agent 数据权限分层
- **范围**：Dev 可参与 schema / migration / 测试数据 / 工程修复；Preview 可读 fixtures 并生成 patch；Prod 只读有限元数据 / 审计 / 监控
- **不做**：Prod 数据写入通道
- **依赖**：M7-03
- **对齐 PRD**：§10.7
- **验收**：Agent 尝试写入 Prod 交易数据或执行高风险 DDL 被拒绝

## M8 · 端到端冒烟（工单 + 报工 + 质检）

### M8-01 工单端到端
- **范围**：从需求输入到首个 Preview 走完整链；度量 §13.1 的 P50 时长

- **范围**：基于工单 Preview 提交真实反馈（文案 + 字段 + 状态机调整各一条），验证三类风险路径
- **依赖**：M5 完成
- **对齐 PRD**：§8.10、§13.1

- **范围**：将质检模块发布到 Prod，再演练一次回滚
- **依赖**：M6–M7 完成

### M8-05 PRD 对齐与 bug bash
- **范围**：按 PRD §14 已明确结论清单做一轮 checklist 对齐；组织一次 1–2 小时的 bug bash；产出 V1.4 候选修订
- **依赖**：M8-01..04
- **对齐 PRD**：§14
- **验收**：checklist 全通过；收集到的 bug 进入正式工单
