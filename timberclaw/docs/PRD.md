### 1.2 术语表

| 术语 | 定义 |
|------|------|
| spec | 描述生成系统的结构化需求说明；**对外**以业务场景视图呈现，**对内**保留结构化专业视图，两者双向映射；版本化存储（详见 §8.2） |
| 业务场景视图 | spec 的面向非技术用户的主视图：以"角色 / 触发 / 主路径 / 异常路径 / 衡量指标"描述一个业务场景，不含实体字段级细节 |

| 主数据 | 不随业务发生而频繁变动的基础数据（物料、BOM、客户、员工、工作中心等），通常由 ERP 等外部系统管理，TimberClaw 通过 CSV / Excel 单向导入使用（详见 §8.12） |

| Dev / Preview / Prod | 三类环境，详见 §9 |
| 数据合同 | 核心业务对象必须提供的最小结构声明（详见 §10.2） |
| 业务使用者（End User） | 真正在日常生产中使用生成系统的工厂人员（排产员 / PMC / 车间 / 质检员 / 班组长等），**不懂代码、不接触 Git 概念**；详见 §4.6 |
| Owner / Reviewer / Admin / Platform Engineer / Human Developer / End User | 六类角色，详见 §4 |

## 2. 产品定义

TimberClaw-code 是一个基于 **OpenHands**（<https://github.com/All-Hands-AI/OpenHands>，MIT 协议）fork 改造而成的工业业务系统 Builder。

> **fork 起始 commit / tag** 由 Platform Engineer 在 M0 启动时以增量修订方式补入本节；建议以验证时的 stable release tag 为锚点，避免在 `main` 上漂移。

它让用户从**自然语言、设计文档和对话**出发，先生成并确认 spec，再在 **Dev** 中自动生成和验证系统，在 **Preview** 中体验和反馈，在人工把关下发布到 **Prod**。

TimberClaw-code 的长期边界不只限于轻量 MES，也可以扩展到采购、库存、质量、设备、看板、流程管理等工业业务系统；但第一阶段至少必须可靠支撑一个轻量 MES 窄业务包。

## 3. 第一阶段目标与非目标

### 3.1 第一阶段目标

第一阶段，Builder 必须能够较稳定地构建出满足以下需求的系统：

- 工单管理
- 报工
- 质检记录
- 基础角色权限
- 基础列表 / 表单 / 详情 / 简单报表页面

这里的“首期业务范围”指的是：**Builder 首期必须具备支撑该业务包的能力**，不是指产品自身只做这个业务包。

### 3.2 非目标

第一阶段不做以下内容：

- 不替代大型重型 MES / ERP / PLM
- 不支持任意技术栈自由生成
- 不支持移动端原生应用
- 不支持复杂多组织、多工厂、多租户模型
- 不支持复杂跨系统主数据治理平台
- 不支持跨系统**实时双向**协同（但支持 **CSV / Excel 单向主数据导入与结果导出**，详见 §8.12）
- 不允许 Agent 自动无审批发布到 Prod
- 不允许 Agent 直接写入 Prod 核心业务数据

### 3.3 首期试点场景：离散制造

第一阶段首个试点场景锁定为 **离散制造**（机加 / 装配 / 电子组装等），对应业务包在 §3.1 基础上细化：

- **工单**：多工序串联；支持派工到班组 / 人 / 设备
- **报工**：工序级报工，支持开工 / 暂停 / 完工
- **质检**：工序级质检，支持合格 / 让步接收 / 返工 / 报废判定
- **基础主数据**：物料、工作中心 / 设备、员工 / 班组；以 §8.12 CSV 单向导入方式维护
- **基础仪表盘**：当班 / 当日产量、工序完工进度、质检合格率（§8.3 一等公民能力）

试点不等于产品只做离散制造；流程制造（化工 / 食品 / 制药）与混合模式作为后续版本可扩展。

## 4. 角色与权限

### 4.1 Owner
**工厂信息化负责人 / IT / 系统管理员**。有基本技术素养（理解"版本""发布""环境"这些概念），但**不是开发者**。在真实工厂场景中，Owner 通常是工厂侧的 IT 主管或信息化专员。

权限：
- 发起需求（也可代业务使用者整理需求）
- 通过**业务场景对话**修订 spec；少数情况下也可直接用结构化专业视图修订
- 确认 spec
- 创建候选版本（面向业务用户呈现为"新建试用版本"）
- 触发 Preview 部署（面向业务用户呈现为"开启在线试用"）
- 查看 Preview 并提交反馈
- 审批高风险 patch
- 输入发布备注并确认 Prod 发布（面向业务用户呈现为"正式启用"）

约束：
- Owner 界面里任何开发者概念（commit SHA、diff、依赖包名、日志堆栈）**必须有人话翻译**，详见 §5.8 和 §7.4
- Owner 不被期望写代码、读日志堆栈；遇到 Dev 验证失败类技术问题，应自动转 Platform Engineer / Admin（见 §8.6）

### 4.2 Reviewer
业务使用者中被指定**代表团队**参与 Preview 试用与反馈的人（常见如班组长、排产主管、质检主管）。

权限：
- 访问 Preview
- 在 Preview 内提交页面体验反馈
- 查看当前候选版本的人话说明（"这是版本 #3，相比版本 #2 调整了……"）

约束：
- 不能触发 Preview 部署、不能确认 spec、不能发布到 Prod
- 不直接看到 Git / rc / commit 等概念（同 §5.8）

### 4.3 Admin
平台管理员 / 系统管理员。

权限：
- 管理环境、项目、权限、发布策略
- 执行或协助 Prod 发布
- 回滚与排障
- 管理数据边界

### 4.4 Platform Engineer
维护模板、底座、集成与工程能力的技术角色。

权限：
- 维护模板底座
- 维护 Builder 运行时
- 维护部署、测试、数据治理与审计机制
- 在 MVP 阶段按冻结式策略维护模板底座

### 4.5 Human Developer
人工介入的开发者角色，仅作为降级与补救通道。

权限：
- 基于 Git 分支 checkout、修改代码、push
- 通过标准 PR 流程参与修复

约束：
- 不单独提供专用 UI 入口
- 不可绕过 Git + PR + Dev 验证流程
- 人工 commit 在合入候选版本前，Agent 必须基于该改动自动生成对应的 spec diff 提案；Owner 未显式确认 spec 同步，该 commit 不能被选作候选版本

### 4.6 业务使用者 (End User)
**真正在日常生产中使用生成系统的工厂人员**：排产员 / PMC / 车间操作工 / 质检员 / 班组长 / 仓管 / 调度等。这是生成系统 **Prod 环境的主要用户群体**，也是 TimberClaw 产品存在的根本原因。

角色特征：
- **不懂代码**、不理解 Git / commit / rc 等概念
- 工作时间大部分在车间现场 / 工位 / 仓库，使用电脑或扫码枪 / 手持终端
- 对"打开就能用、填表不出错、少点两下"的敏感度远高于对"系统漂亮"的敏感度
- 对停机 / 数据错 / 报工对不上账的容忍度极低

权限：
- **仅** Prod（和被升为 Reviewer 时的 Preview）
- 按 spec 中定义的角色访问对应页面 / 动作
- 默认不看见 Builder 侧任何入口

约束：
- 业务使用者**不直接**接触 Builder（不访问需求输入 / spec / 候选版本 / PR / 审计日志这些页面）
- 他们对 Builder 的存在感知应当接近"每隔一段时间系统多了一个功能"，不会被"版本""试用环境"这类概念干扰
- 业务使用者的反馈链路是：口头 / 线下 → 被 Owner 或 Reviewer 整理后录入 Builder（MVP 范围）
- 生成系统为业务使用者准备的所有页面必须满足 §5.7 工业风格和 §8.3 模板底座中面向 End User 的约束（高信息密度、操作导向、错误提示清晰）

## 5. 核心产品原则

### 5.1 Spec-first
用户可以没有现成 spec 开始，但系统必须先生成一个初始 spec。

用户必须能够通过**两种方式**修订 spec：
1. **自然语言对话修订**
2. **结构化 spec 编辑修订**

代码生成前必须有一次 Owner 的**明确 spec 确认**。

### 5.2 Preview-first 验收
PR 是工程载体，Preview 是业务体验载体。业务验收主要发生在 Preview，而不是代码仓库。

### 5.3 三环境强分离
- **Dev**：可随时运行，用于生成、验证、自动修复
- **Preview**：需要 Owner 明确触发，从 PR / 候选版本部署，用于体验、评论、验收
- **Prod**：只能在人工确认后发布

### 5.4 受约束生成
Agent 必须在受约束的模板、目录结构、组件体系和数据规范内工作，不允许从空白仓库自由生长。

### 5.5 数据安全优先于生成速度
Prod 数据稳定性、数据模型一致性、审计可追溯性和写入边界优先级高于自动化速度。

### 5.6 最小化原则
这个系统本身的代码，以及它生成出来的系统代码，都必须在满足功能要求的前提下尽量精简。

精简的含义包括：
- 模块数量尽量少
- 依赖尽量少
- 抽象层级不过度设计
- 优先复用统一模板和组件
- 不为了未来假设需求而提前引入复杂机制
- 生成系统的页面、表、接口、流程都只包含当前需求所必需的部分

### 5.7 工业风格
TimberClaw-code 及其生成系统的默认设计风格应适用于工业场景，要求：

- **简约**
- **大气**
- **稳定**
- **可读**
- **操作导向**

默认风格约束：
- 避免花哨动效和娱乐化视觉
- 优先高信息密度下的清晰布局
- 色彩中性克制，状态色清晰但不过度鲜艳
- 列表、表单、状态标签、操作区应清楚直接
- 以可靠、专业、耐看的工业工作台风格为默认

### 5.8 概念隔离：工程域概念不得泄漏到业务用户界面

Builder 本体承载大量工程域概念（commit、PR、rc-N、diff、CI、日志、lint、migration、依赖包名、堆栈）。这些概念对 Owner 部分可见、对 Platform Engineer / Admin 全可见，但**对业务使用者 / Reviewer 不可见**。生成系统中尤其不能出现。

硬规则：

1. 在**生成系统**（TimberClaw 输出给业务使用者的系统）的任何页面、文案、提示、错误信息中，**禁止**出现：commit SHA、PR 编号、rc 编号、Git 分支名、"环境"这个词以其英文形式、堆栈、依赖名
2. 在 Builder 面向 Owner 的业务屏（需求输入、spec、候选版本、Preview、发布、反馈）中：
   - 必须优先使用 §7.4 语言映射表里的人话术语
   - 工程细节允许通过"展开详情"、"查看技术信息"这类次级入口展示，供 Platform Engineer / Admin 使用
3. 在 Builder 面向 Platform Engineer / Admin 的屏（模板底座、Dev 验证日志、LLM 监控、账号管理）里，可使用原始工程术语，不受本条约束
4. Reviewer 看到的候选版本说明必须是自然语言摘要（例如"调整了质检页的合格判定，增加了返工类型"），不允许出现 diff / commit 级别信息

判定准则：如果一个文案 / 标签让一个从未接触过代码的工厂调度员困惑，就视为违反本条原则，必须走 §8.10 低风险 patch 修订。

## 6. 默认实现边界

为避免 Coding Agent 因需求不清而停滞，第一阶段采用以下默认值。

### 6.1 技术栈（第一阶段冻结）

| 层 | 选型 | 说明 |
|----|------|------|
| 前端框架 | **React 18 + TypeScript**（强制 TS） | 与 OpenHands 上游保持一致，降低合并成本 |
| 前端 UI 组件库 | **Ant Design v5** | 工业 / 企业后台事实标准；业务屏可按需引入 Ant Design Pro 的 ProTable / ProForm / ProLayout |
| 构建工具 | Vite | 与 OpenHands 对齐 |
| 后端框架 | Django（LTS） | 自带 Auth、Admin、ORM、Migration，对 MVP 范围足够 |
| ORM | Django ORM | 不额外引入 SQLAlchemy 等 |
| Migration 工具 | Django Migrations | 不额外引入 Alembic 等 |
| 数据库 | PostgreSQL | 唯一关系型数据库 |
| 代码托管 | GitLab（自建 / 内网） | 所有代码不出内网 |
| Preview / Prod 部署 | Docker Compose | 单节点多服务，Preview 与 Prod 使用同一部署形态以降低差异 |
| 包管理 | 前端 pnpm / 后端 pip + `requirements.txt`（或 `pyproject.toml`） | 可复现构建 |

变更规则：
- 技术栈增删视为模板底座变更，必须经 Platform Engineer 评审
- Agent 不得自作主张引入新框架或组件库
- 任何不在本表中的依赖，引入时必须在 PR 描述中说明理由

### 6.1.1 Builder 前端 UI 一致性策略（务实路线）

为最小化与 OpenHands 上游的合并冲突：

- **继承 OpenHands 的屏**（与 Agent 对话、代码 diff 查看、文件树、终端等开发者向场景）：沿用 OpenHands 现有 UI 风格与 Tailwind 样式，**不强行改写为 AntD**
- **TimberClaw 新增的业务屏**（需求输入、spec 编辑器、候选版本、Preview 管理、发布审批、反馈收件箱）：统一使用 **Ant Design v5**
- 两套风格共存，但以**清晰的路由前缀 / 布局分层**区隔；公共导航 / 顶栏做一次视觉贴合
- **生成系统（TimberClaw 的输出物）**：一律使用 React 18 + TypeScript + Ant Design v5，与 Builder 的业务屏保持一致

### 6.2 认证与账号（MVP 最小子集）

- Builder 自身登录：Django 内置 auth + 用户名 / 密码，密码采用 Django 默认哈希策略
- 生成系统的登录：同样基于 Django 内置 auth，与 Builder 账号体系**相互独立**
- 第一阶段不做 SSO / OAuth / LDAP，仅在代码层面预留扩展位
- 第一阶段仅实现"角色 ↔ 动作"级权限，角色固定为 §4 列出的五类，不支持自定义角色
- 账号新增 / 禁用由 Admin 通过 Django Admin 或等效后台完成

### 6.3 其他默认约束

1. **仅支持 Web 系统**
2. **单项目单应用**
3. **单一标准工程底座**
4. **Git + PR 驱动**
5. **Spec 必须版本化**
6. **系统生成前必须有一次明确的 spec 确认**
7. **Preview 必须由 Owner 明确触发，不自动跟随每次 Dev 修改**
8. **Prod 必须人工确认后发布**
9. **Preview 不得直连正式生产库**
10. **Agent 不得直接改写 Prod 数据或执行高风险 Prod DDL**
11. **第一阶段默认不做双人审批**（Prod 发布由 Owner 单人确认，不强制背靠背）
12. **第一阶段默认不做复杂多租户**
13. **第一阶段默认不做通用可视化流程编排器**
14. **第一阶段默认单体优先，不拆微服务**
15. **第一阶段默认不做跨系统同步实现，仅保留未来扩展基础**

### 6.4 工厂环境现实约束已知项（MVP 范围）

工厂场景的典型现实约束，MVP 的默认策略如下：

| 约束 | MVP 默认策略 |
|------|-------------|
| 网络间歇性不稳定 | 生成系统的关键录入页（报工 / 质检）必须具备失败重试提示与"未保存"显式状态，避免静默丢数据；MVP 不做离线模式 |
| 扫码 / 条码录入 | 关键录入字段（工单号、物料号、员工工号）允许由扫码枪直接输入；统一规范"扫码触发回车"提交行为；不专门接硬件 SDK |
| 打印需求 | MVP 通过浏览器原生打印支持单据 / 标签打印；不集成打印机驱动 / ZPL 等 |
| 手持 / 平板使用 | 生成系统页面需在 **≥ 10 英寸平板** 横屏下可用；**不**适配 5~6 英寸手机 |
| 多车间 / 多工序数据 | 通过"工作中心""车间"这类维度字段做**软隔离**（按角色过滤），不做独立库 / 独立部署 |
| 数据对账 | 关键交易（报工、质检、派工）必须有唯一业务键；导入失败行可下载重录，不静默忽略 |

本节内容是 Builder 在生成系统时**默认采用**的处理方式；具体业务包可在 spec 里推翻或加强，但 Agent 不得主动偏离。

## 7. 版本与环境流转

### 7.1 候选版本定义

候选版本是：**某个 PR 内一个被 Owner 明确选定的 commit SHA，并赋予递增的 `rc-N` 编号**。

MVP 中，候选版本的产品语义为：
- 来源于单个 PR
- 绑定到该 PR 中一个确定的 commit
- 由 Owner 创建
- 以 `rc-1`、`rc-2` 等方式展示和追踪

说明：
- 为保持最小化，MVP 不引入 release branch
- `rc-N` 是候选版本编号，**按 PR 内部递增**（每个 PR 的候选版本从 `rc-1` 开始），不要求底层必须依赖 GitLab 原生的 commit label / release 能力；系统内部只需能稳定记录 `PR + commit SHA + rc-N`
- 候选版本一经创建不再复用编号；Owner 可将某个 `rc-N` 标记为废弃（不再触发新的 Preview / Prod 发布），但编号保留

### 7.2 主流程

需求输入
→ 初始 spec 自动生成
→ 用户通过对话或结构化编辑修订 spec
→ Owner 确认 spec
→ 代码生成
→ Dev 验证
→ 自动创建 / 更新 PR
→ Owner 选择 PR 中某个 commit 创建候选版本（rc-N）
→ Owner 触发 Preview 部署
→ 人工体验与评论
→ 自动或人工确认后的 patch
→ 更新 PR
→ Owner 重新选择候选版本并触发新的 Preview（如需要）
→ Owner 输入发布备注并确认
→ Prod 发布

### 7.3 流程规则

1. 没有 spec 时，系统必须自动生成初始 spec
2. 有 spec 时，也允许系统做规范化整理
3. 代码生成前必须有一次明确 spec 确认
4. Dev 可随时执行
5. Preview 只能从候选版本触发
6. 候选版本必须来源于 PR
7. Preview 必须由 Owner 明确触发
8. Prod 只能从已完成 Preview 验收的候选版本发布
9. Prod 发布必须人工确认且必须填写发布备注

### 7.4 面向业务用户的产品语言映射（强制）

Builder 面向 Owner / Reviewer 的 UI、以及生成系统里所有用户可见文案，都必须按下表做语言映射。开发者术语只允许出现在 Platform Engineer / Admin 专区（见 §5.8）。

| 工程概念 | Owner UI 用词 | Reviewer / 生成系统 UI 用词 |
|----------|----------------|-------------------------------|
| commit SHA | （隐藏） | （隐藏） |
| PR / Merge Request | "修改集 #n" | （隐藏） |
| 候选版本 rc-N | "版本 #N / 试用版 N" | "版本 N" |
| Preview 部署 | "开启在线试用 / 试用环境" | "试用系统" |
| Prod 发布 | "正式启用 / 上线 版本 #N" | "系统更新" |
| Dev 验证失败 | "系统自检发现问题" | （隐藏；由 Owner 处理） |
| Agent 自动 patch | "系统自动修改" | （隐藏） |
| migration | "数据结构调整" | （隐藏；仅必要时简述"数据字段更新中"） |
| 回滚 | "撤回到上一个版本" | "系统已还原为上一个版本" |
| spec diff | "需求变化摘要" | "版本变化说明" |

实现要求：术语表集中维护在 Builder 前端，不允许在各屏里硬编码英文工程词。

## 8. 功能模块

### 8.1 需求输入中心

#### 目标
接收用户需求，并在用户没有 spec 时自动生成初始 spec 草稿。

#### 输入方式
- 上传设计文档
- 文本输入需求补充
- 直接对话描述需求
- 选择基础业务模板（如工单、报工、质检）

#### 主要输入者
- **Owner** 是 Builder 的主要输入者
- **业务使用者**（排产员 / PMC / 质检等）的诉求通常通过**口头 / 线下访谈**汇集到 Owner，由 Owner 整理后录入；MVP 不为业务使用者单独提供需求提交入口
- 允许 Owner 以"代写"口吻描述某位业务使用者的原话，Agent 应识别并保留这种引述关系，便于后续 spec 呈现为业务场景视图（§8.2）

#### 输出
- 原始需求包
- 需求摘要
- 待确认问题列表
- 初始 spec 草稿

#### 必需能力
- 保存原始需求
- 自动生成需求摘要
- 自动生成初始 spec
- 标出不明确项
- 保存需求版本

### 8.2 Spec 管理

#### 目标
将非结构化需求转化为结构化 spec，并支持用户持续修订。

#### spec 双视图（V1.5 引入）

Spec 在存储与处理上是**同一份结构化数据**，但在 UI 上必须同时提供两个视图，默认入口为业务场景视图：

**业务场景视图（主视图，面向 Owner，默认打开）**
- 以"业务场景卡片"为组织单元；一个业务场景 = {角色、触发、主路径、异常路径、衡量指标}
- 例：`质检员发现不合格 → 触发返工 → 返工工单自动串接到原工单下 → 统计：返工率`
- 支持自然语言对话直接修订
- 隐藏字段 / 实体 / 状态枚举等技术细节；但可展开"涉及的数据与页面"次级面板查看（只读）

**结构化专业视图（辅助视图，面向 Platform Engineer / Admin，默认折叠）**
- 展开 spec 内部的完整结构：实体、字段、页面清单、状态机、权限矩阵、校验规则、报表 / 仪表盘定义、集成
- 支持结构化编辑（字段增删改、页面清单调整、角色矩阵修改）
- 仅 Owner 可主动切换进入；Reviewer 不可访问
- 专业视图的改动必须能反推出对业务场景视图的摘要说明（见 §7.4 spec diff 行）

#### spec 最小内容（结构化专业视图中呈现的底层结构）
- 应用信息
- 实体与字段
- 页面清单（包含**列表 / 表单 / 详情 / 仪表盘**四类页面，仪表盘见 §8.3）
- 角色与权限
- 流程 / 状态机
- 校验规则
- **报表 / 仪表盘定义**（数据源、指标、维度、图表类型）
- **主数据来源声明**（哪些实体以 CSV / Excel 导入维护，哪些本系统内维护，见 §8.12）
- 集成需求（如有）
- 环境要求

#### 修订方式
必须同时支持：
1. **业务场景视图 + 自然语言对话修订**（Owner 默认路径）
2. **结构化专业视图 + 表单编辑修订**（Platform Engineer / Owner 按需使用）

#### 必需能力
- 自动生成初始 spec（默认即生成业务场景视图）
- 展示 spec diff：**两种视图各自一份**（业务场景视图给 Owner 看，结构化专业视图给 PE / Admin 看）
- 保存 spec 版本
- 显式确认当前 spec（Owner 必须在业务场景视图下确认；确认行为等价于冻结两个视图）
- 将业务逻辑类变更优先回写 spec（代码反推 spec 时也必须生成业务场景视图层的摘要）

### 8.3 Builder 模板底座

#### 目标
基于统一模板底座生成最简可用系统。

#### 第一阶段固定模板底座
在 §6.1 技术栈之上，模板底座内置以下通用能力：

- 用户 / 角色 / 权限（基于 Django 内置 auth）
- 实体 CRUD（Django Model + DRF 或等价 JSON 接口层）
- 状态流转
- 操作日志
- **四类一等公民页面**（React 18 + TypeScript + Ant Design v5 通用模板）：
  - 列表页（默认 ProTable）
  - 表单页（默认 ProForm）
  - 详情页
  - **仪表盘页（一等公民，详见下方 §8.3.1）**
- **主数据导入 / 导出能力**（CSV / Excel 单向导入 + 结果 CSV 导出，详见 §8.12）
- PostgreSQL 连接与 Django Migrations
- 基础测试能力（后端 pytest-django / 前端 Vitest + React Testing Library）
- Preview / Prod 部署适配层（Docker Compose 模板 + 环境变量管理）

#### 8.3.1 仪表盘作为一等公民

背景：MES / 工厂业务的核心产出物之一是**看数**（当班产量、工序 OEE、质检合格率、返工率等）。如果仪表盘只是"简单报表"的附属物，生成的系统就不能满足最基本的工厂管理诉求。因此 V1.5 起把仪表盘提升为与列表 / 表单 / 详情并列的页面类型。

**必需能力**：

- 模板底座内置一个**仪表盘页面模板**，基于 **Ant Design Charts（或 AntV G2Plot）**；不额外引入 ECharts 等重型依赖
- 支持以下图表类型（MVP 范围）：柱图、折线图、饼图 / 环形图、数据看板卡片（KPI 数字 + 环比）、明细表
- 每个仪表盘页面由 spec 声明：
  - 数据源（绑定到某实体或其聚合 API）
  - 指标定义（计数 / 求和 / 均值 / 合格率 / 时间段内增长率）
  - 维度（按日 / 按班次 / 按工作中心 / 按物料 / 按工单状态……）
  - 过滤条件（角色过滤、日期过滤、状态过滤）
  - 图表类型
- 支持点击图表某项**下钻到明细**（下钻跳转到对应列表页，带上维度过滤）
- 数据查询以后端聚合接口实现，不把原始明细拉到前端再算

**约束**：

- 仪表盘不做拖拽式设计器（MVP 不做 §12.3）
- 仪表盘的指标、维度、图表类型**来源于 spec**，由 Agent 根据场景生成；业务使用者不能在运行时自由定义指标
- 仪表盘页面查询必须有 SLA：单次打开 Prod 数据量 ≤ 100 万行时 P95 响应 ≤ 3 秒（不达标由 Platform Engineer 通过索引 / 预聚合优化，而不是换框架）

#### 模板底座演进策略
- MVP 阶段由 **Platform Engineer** 维护
- 采用**冻结式演进**
- 只修 bug，不主动加新能力，除非当前业务包必需
- 每次模板变更都必须验证已有生成系统的 Dev 验证是否仍通过
- 模板底座不独立发版，跟随产品版本演进

#### 约束
- 只维护一套标准脚手架
- 不为首期以外需求引入额外框架
- 生成系统优先在该底座内扩展，不新开平行体系

### 8.4 系统生成

#### 目标
根据 spec 和模板底座生成最简可用系统。

#### 生成要求

- 不生成未被当前需求使用的附加模块
- 不引入无明确业务理由的新抽象
- 默认采用工业场景的简约大气设计风格

#### 第一阶段必须支撑的业务对象
- 工单
- 报工记录
- 质检记录
- 用户 / 角色 / 权限

#### 第一阶段必须支撑的业务能力
- 基础 CRUD
- 基础状态流转
- 基础角色权限
- 基础操作日志

### 8.5 Agent Orchestrator

#### 目标
驱动系统完成从 spec 到代码、验证、PR、部署、修复的闭环。

#### 必需角色
1. **Spec Analyst**：从需求生成和修订 spec
2. **App Architect**：基于 spec 产出实现计划
3. **Code Builder**：在模板底座内改代码
4. **Verifier**：跑 Dev 验证
5. **Deploy Operator**：部署 Preview / 协助发布
6. **Repair Agent**：根据错误和反馈生成 patch

#### 约束
- 每一步状态可追踪
- Agent 修改范围受限
- Agent 不得绕过 Git / PR 流程
- Agent 不得直接对 Prod 做高风险操作

#### LLM 与执行底座
- 第一阶段允许调用云端 LLM API（例如 OpenAI / Anthropic / 国内主流厂商），但 **Builder 本体必须部署在本地内网**
- LLM 供应商与模型通过**配置切换**，代码层面不得硬编码具体厂商 SDK；必须抽象一层统一的 LLM Gateway
- 必须提供 Token 用量、请求成功率、失败原因的基础监控
- 必须设置单任务最大 Token / 最大时长上限，超限自动终止并上报
- 所有发往云端 LLM 的请求经由统一出口并留审计；敏感字段（凭据、内网主机名、真实业务数据）在出站前必须过滤或脱敏

---

### 8.6 Dev 验证

#### 目标
在开发态自动验证代码可用性，并修复明确的工程错误。

#### 必需验证项
- lint
- type check
- build
- migration 检查
- 基础测试

#### 规则
- Dev 可随时触发
- 对确定性工程错误可自动修复
- 高风险问题应终止并转人工
- Dev 成功是进入候选版本与 Preview 的前置条件之一

#### 代码层面"高风险"问题的认定
满足以下任一条件即视为高风险，Agent 必须停止自动修复并转人工：
- migration 含 §10.3 定义的任何高风险 schema 变更
- 测试未能覆盖新增 / 变更的业务分支
- 涉及认证、授权、加密、审计模块的改动
- 新增一级依赖（前端 `package.json` 顶层依赖、后端 `requirements.txt` / `pyproject.toml` 顶层依赖）
- 模板底座自身文件的改动
- 连续两次自动修复仍未通过 Dev 验证

#### "转人工"的具体接盘角色

"转人工"在 TimberClaw 里**不等于**把问题抛给 Owner，因为 Owner 是工厂 IT，不负责读堆栈。默认接盘顺序：

1. 首要接盘：**Platform Engineer**。所有 Dev 验证失败、依赖冲突、migration 报错、模板底座异常、LLM Gateway 故障默认先转 Platform Engineer
2. 次要接盘：**Admin**。当 Platform Engineer 判定问题与工程无关而是账号 / 环境配置 / 权限配置导致时，转 Admin
3. Owner 被通知、但不是接盘人：Owner 仅在一种情况下必须决策：Agent 判定"继续修"需要 Owner 同意修改 spec 范围（例如字段定义分歧）
4. 生成系统的业务使用者在本节流程中**完全不可见**

Builder UI 必须在"转人工"时按如下规则分派通知：给 Platform Engineer 的通知使用原始工程术语；给 Owner 的通知必须用 §7.4 语言映射后的人话（"系统自检发现问题，工程师已介入处理"即可）。

### 8.7 Git 与 PR 管理

#### 目标
将系统生成与迭代纳入标准研发流程。

#### 平台与接入
- 使用自建 / 内网 **GitLab** 作为唯一代码托管平台
- Builder 通过 **GitLab Project Access Token**（或等效 Deploy Token）与 GitLab 交互，令牌仅具备对目标仓库的最小必要权限（`read_repository` + `write_repository` + `api`）
- Agent 相关 commit 使用统一的机器账号身份，便于审计过滤

#### 分支与命名
- 默认主分支：`main`
- Agent 生成分支命名规则：`tc/<spec_version>/<short_desc>`
- 禁止直接向 `main` 推送
- PR 在 GitLab 中以 Merge Request（MR）形式存在；本 PRD 中"PR"均指 GitLab MR

#### 必需能力
- 自动创建分支
- 自动生成 commit
- 自动创建 PR（MR）
- 后续 patch 持续更新 PR
- 记录 PR 对应的 spec 版本
- 记录候选版本对应的 commit SHA 与 rc-N

#### 规则
- 不允许直接改主分支
- 所有自动修改必须留痕
- PR 是候选版本与 Preview 的唯一来源

### 8.8 Preview 部署与体验

#### 目标
提供供业务人员体验与验收的候选系统环境。

#### 必需能力
- 每个候选版本可对应一个 Preview
- 提供 Preview URL
- 展示部署状态与日志
- Preview 部署时自动加载样例数据 fixtures

#### 部署形态
- 每个 Preview 是一组独立的 **Docker Compose** 服务（应用 + PostgreSQL + 可选后台服务）
- Preview 运行在本地内网的 Preview Runner 节点上
- Preview 之间完全隔离，各自使用独立的 PostgreSQL 实例或独立数据库 schema

#### 生命周期
- 同一项目允许并存多个 Preview，数量上限由 Admin 在项目配置中声明
- Preview 默认保留期为 **7 天**，到期由平台自动回收；Owner 可在 Preview 页面延长
- 候选版本被废弃时，其对应 Preview 立即回收
- Preview 部署失败允许自动重试 1 次，仍失败则标记为失败并通知 Owner

#### 权限规则
- **仅 Owner 可触发 Preview 部署**
- Reviewer 可访问 Preview，但不能触发部署
- Admin 可在排障场景下协助处理，但不改变 Owner 触发的主规则

#### 样例数据策略
- 由 Code Builder 根据 spec 自动生成 fixtures
- fixtures 至少包含 3–5 条典型数据，覆盖主要状态
- Preview 部署时自动加载 fixtures
- MVP 不做脱敏生产数据导入

#### 其他规则
- Preview 不自动跟随每次 Dev 修改
- Preview 必须来自候选版本
- Preview 不得直连正式生产库

### 8.9 反馈采集

#### 目标
把用户在 Preview 中的意见转化为可执行的修改输入。

#### MVP 交互方式
采用：**独立反馈面板 + 页面路径自动捕获**。

具体规则：
- Preview 页面提供统一反馈入口
- 用户点击反馈按钮后打开独立反馈面板
- 系统自动带入当前页面路径 / Preview URL
- 用户选择反馈类型并填写描述
- MVP 不做页面内自由标注
- MVP 不强依赖组件标识；如可获取，可作为附加信息记录

#### 反馈类型
- 文案问题
- 页面布局问题
- 字段设计问题
- 流程问题
- 权限问题
- 功能 Bug
- 报表问题

#### 必需数据
- Preview URL / 页面路径
- 反馈类型
- 反馈内容
- 提交人
- 关联候选版本
- 组件标识（如可获取）

### 8.10 反馈理解与自动 patch

#### 目标
根据反馈判断应修改 spec 还是代码，并生成后续 patch。

#### patch 分层策略

##### 低风险修改
包括：
- 文案
- 展示顺序
- 简单布局
- 筛选展示
- 轻量交互文案

处理策略：
- 可自动 patch
- 进入 PR
- 重新走 Dev 验证
- 后续如需新的 Preview，仍由 Owner 明确触发

##### 高风险修改
包括：
- 数据模型
- 状态机
- 权限边界
- 影响核心业务语义的流程变化

处理策略：
- 必须先人工确认
- 优先修改 spec
- 再驱动代码变更
- 必须进入 PR
- 重新走 Dev 验证
- 后续 Preview 仍由 Owner 明确触发

#### 风险判定
判定方式采用**静态规则 + Agent 研判**双保险：
- 触达以下任一路径 / 改动类型的 patch **强制**标为高风险：
  - migration 文件
  - 认证、授权、审计相关模块目录
  - 状态机定义
  - 模板底座相关文件
- 其余情况由 Agent 给出风险等级，Owner 可在反馈面板中人工升 / 降级
- 若事后发现某低风险 patch 实际触及高风险范围，必须立即回滚该 patch 并记录为违规事件，作为后续规则校准的输入

#### 总规则
- 业务逻辑、数据模型、流程定义类反馈：优先修改 spec，再驱动代码变更
- 纯展示、工程修复类反馈：可直接 patch 代码
- 所有 patch 必须进入 Git / PR

---

### 8.11 Prod 发布

#### 目标
将经过 Preview 验收的候选版本发布到正式环境。

#### 必需能力
- 查看候选版本
- 查看 PR、spec 版本、验证状态
- 查看审计信息
- 输入发布备注
- 人工触发发布
- 支持回滚

#### 规则
- Prod 发布必须建立在已验收的候选版本之上
- Preview 通过不等于自动发布
- Prod 必须人工确认后发布
- 发布备注必填
- Prod 数据库与配置必须独立管理

#### 回滚
- 回滚对象为"上一个在 Prod 生效过的 rc"，不允许跳版回滚
- 回滚必须由 Admin 或 Owner 发起并填写回滚备注
- 涉及不可逆 migration（如删字段、改字段类型）时，回滚需由 Platform Engineer 人工介入并确认数据处理方案
- 回滚不等于"撤销 PR"，原 PR 与 rc 记录保留
- 回滚操作进入同一审计流（来源标记为 `rollback`）

### 8.12 主数据导入与结果导出

#### 目标

工厂 MES 不是一座孤岛。它至少需要能**消费**来自 ERP / 人事 / 工艺系统的主数据（物料、BOM、客户、员工、工作中心），以及把关键结果（报工明细、质检明细、当日产量）以标准格式交回。

MVP 不做实时双向集成，以 **CSV / Excel 文件方式**满足最小集成闭环。

#### MVP 必需能力

**主数据导入**

- 模板底座为每个被声明为"外部主数据"的实体自动生成 CSV / Excel 导入页
- 每个实体必须有一份 **导入模板**（列名 + 必填标记 + 示例值），可下载
- 上传时先做**预校验**：列名匹配、必填校验、枚举值校验、业务键唯一性
- 校验失败行不入库，提供"失败行下载"（保留原始数据 + 错误原因列）
- 成功导入行必须幂等：**同业务键 upsert**，不重复创建
- 导入操作进入操作日志（谁导入 / 何时 / 影响行数 / 失败行数）
- MVP 不做定时 / 接口自动导入，只做人工触发上传
- 每个实体在 spec 里声明以下属性之一：`internal`（本系统内维护）/ `external_import`（外部导入）/ `mixed`（允许两种）；Agent 在生成 CRUD 页时据此决定是否允许人工新增 / 编辑

**结果导出**

- 列表页必须默认支持**按当前筛选条件导出 CSV**
- 仪表盘必须支持把当前查询结果（明细 + 聚合）导出 CSV
- 导出不暴露原始 ID（SHA / 内部主键）；业务键必须出现
- 导出文件名规则：`<entity>_<yyyymmdd>_<hhmm>.csv`，必须 UTF-8 + BOM（保证中文 Excel 打开不乱码）

#### 非目标（MVP 不做）

- 与 ERP / PLM 的实时接口对接
- 主数据变更推回 ERP
- 复杂主数据治理、版本管理
- 自定义字段映射器（列映射统一以"列名精确匹配"为准）

#### 约束

- 任何被标为 `external_import` 的实体，其本系统内 CRUD 页**不允许**提供"新增 / 批量删除"按钮；只允许查看、筛选、导出
- 主数据导入不得触及业务交易表（工单、报工、质检记录）：这些是"本系统内维护"的核心对象

## 9. 环境设计

### 9.1 Dev
用途：生成、验证、自动修复

特点：
- 可随时运行
- 可频繁重建
- 数据可重置
- 仅供 Agent / 平台使用

### 9.2 Preview
用途：体验、评论、验收

特点：
- 来自候选版本
- 需 Owner 明确触发
- 尽量接近正式运行形态

### 9.3 Prod
用途：正式业务运行

特点：
- 独立数据库
- 独立配置
- 严格受控
- 可审计、可回滚

### 9.4 Builder 自身部署

- MVP 阶段 Builder 本体部署在用户内网，与 GitLab、Preview Runner 同处同一网络
- 使用 Docker Compose 运行 Builder 应用、元数据 PostgreSQL、Agent 执行 Worker
- Builder 的升级走常规研发流程（Git + PR + 发布），与生成系统的发布流程相互独立
- 允许 Builder 通过显式配置的出站代理访问云端 LLM（见 §8.5）；除此之外不得访问公网
- 长期不排除提供多租户 SaaS 形态，但 SaaS 相关能力（账号体系升级、数据隔离、计费等）均不纳入 MVP

### 9.5 审计日志保留与备份（MVP 简化版）

- 操作日志保留期 ≥ 180 天
- Prod 数据库至少每日一次全量备份，保留期 ≥ 30 天
- 备份与日志应存放在与业务库分离的存储路径
- MVP 不要求异地 / 多副本；允许通过 cron + 本地挂载卷实现

## 10. 数据体系与安全治理（MVP）

业务系统的数据安全，不只是防泄漏，而是确保数据模型稳定、写入受控、变更可追溯、环境隔离清晰，并且 Agent 不能绕过这些约束直接影响 Prod。

### 10.1 数据建模原则

数据体系分四层：
1. **业务对象层**：工单、报工、质检等对象定义
2. **业务规则层**：合法状态流转、字段约束、不可变字段
3. **逻辑数据模型层**：实体、字段、主外键、约束、索引
4. **物理存储层**：数据库表

#### 关键规则
- 先定义业务对象与规则，再落表
- 核心对象必须有唯一标识规则
- 核心对象必须有审计字段
- 核心对象必须定义状态机或业务约束

### 10.2 数据合同（MVP 简化版）

MVP 中，每个核心对象必须定义最小数据合同，仅包括：
- 主键定义
- 业务键定义
- 字段清单
- 状态枚举定义

#### 关键规则
- 没有最小数据合同的核心对象，不允许进入 Prod
- MVP 不要求 owner system 和同步策略字段

### 10.3 Schema 变更规则

#### 关键规则
- Agent 不得直接修改 Prod schema
- 所有 schema 变更必须走 migration
- 高风险变更必须人工审批
- 优先采用前向兼容方式演进

#### 高风险变更包括
- 删字段
- 改字段类型
- 改主键 / 唯一约束
- 拆表 / 合表
- 修改关键状态枚举语义

### 10.4 数据写入边界

#### 关键规则
- Prod 数据写入必须通过受控 API / 服务路径
- 不能通过页面脚本或 Agent 直接乱写核心表
- 状态机规则必须在服务层生效，不只依赖前端
- 权限至少控制到角色级和动作级

### 10.5 审计与追溯（MVP 简化版）

MVP 只要求基础操作日志，记录：
- 谁执行了什么动作
- 动作时间
- 作用对象
- 修改来源

来源至少区分：
- human_ui
- agent_patch
- migration_job

MVP 不强制要求改前值 / 改后值的全量字段级审计。

### 10.6 环境隔离、恢复与备份（MVP 简化版）

#### 关键规则
- Dev / Preview / Prod 数据库必须隔离
- Preview 不得直连正式库
- Prod 必须具备基础备份能力
- migration 前必须有可恢复手段
- 核心业务对象优先软删除或保留历史

MVP 中：
- Prod 备份可通过简单定时任务完成
- 不要求细粒度 point-in-time 恢复

### 10.7 Agent 数据权限

#### Dev
可参与 schema 草案、migration 草案、测试数据、工程修复

#### Preview
可读取 fixtures 并生成 patch

#### Prod
默认只能读有限元数据、审计信息、监控信息；不得直接写正式交易数据、不得直接执行高风险 DDL

- 业务使用者在 Builder 侧的独立账号或入口

## 13. 成功标准

以下指标为 MVP 基线，后续版本可逐步收紧。

### 13.1 产品效率
- 从首次需求输入到首个可用 Preview（含 spec 确认 → 代码生成 → Dev 验证 → rc-1 创建 → Preview 部署）的 P50 时长 ≤ 30 分钟
- 从反馈提交到 Agent 给出 patch 的 P50 时长 ≤ 10 分钟
- 首次代码生成一次性通过 Dev 验证的比例 ≥ 60%

### 13.2 可用性
- 能稳定支撑离散制造"工单 + 报工 + 质检 + 基础仪表盘"系统端到端构建
- 业务人员可在 Preview 中完成主要验收，无需回仓库查代码
- 高风险修改 100% 经过人工确认
- **场景达成率**：首期试点 §3.3 的 5 个业务场景（工单创建→派工→报工→质检→仪表盘看当日产量），首次 spec 确认后生成系统，一次性跑通 ≥ 4 个；剩余 1 个允许走 1 轮 patch 闭环内修复
- **概念隔离达标率**：Owner / Reviewer / 生成系统 UI 中出现未经 §7.4 映射的原始工程词的情况 ≤ 5 处 / 版本

### 13.3 数据安全
- Prod 数据结构稳定，不出现未经 migration 的 schema 改动
- Preview / Prod 数据库 100% 隔离
- 关键动作操作日志覆盖率 ≥ 95%
- Agent 无法绕过 Prod 写入边界（以审计日志抽检 0 例违规为准）

### 13.4 工程质量
- Dev 验证通过率（含自动修复后）≥ 80%
- Preview 部署成功率 ≥ 90%
- Prod 发布失败后回滚时长 ≤ 15 分钟

## 14. 已明确结论

以下内容已明确，不再视为待定项：

1. 项目名称为 **TimberClaw-code**
2. 产品边界不只限于轻量 MES，但第一阶段至少要支撑 MES 类需求
3. 第一阶段必须能支撑“工单 + 报工 + 质检”窄业务包
4. 用户没有 spec 也可以开始
5. 系统必须自动生成初始 spec
6. spec 修订同时支持**自然语言对话**和**结构化编辑**
7. spec 确认后才能进入代码生成
8. Dev 可随时运行
9. Preview 由 **Owner** 明确触发，且从候选版本过去
10. 候选版本定义为 **PR 内特定 commit SHA + rc-N 编号**
11. 评论后的自动 patch 采用**低风险自动、高风险人工确认**的分层策略
12. 首期 Builder 采用固定的“系统模板底座”
13. 模板底座在 MVP 阶段采用冻结式演进
14. 整体设计风格适用于工业场景，要求**简约大气**
15. Prod 需要人工确认后发布，且必须填写发布备注
16. MVP 反馈交互采用**独立反馈面板 + 页面路径自动捕获**
17. MVP 样例数据由 Agent 根据 spec 生成 fixtures，并在 Preview 部署时加载
18. 人工开发者可介入，但必须复用 Git + PR 流程，不单独建设专用通道
19. MVP 数据治理采用最小子集：做建模、最小数据合同、schema 规则、写入边界、基础操作日志、隔离与基础备份、Agent 数据权限
20. 这个系统本身和它输出的系统都必须遵循最小化原则
21. 第一阶段技术栈冻结为：**React 18 + TypeScript + Ant Design v5**（前端）、Django + Django ORM + Django Migrations（后端）、PostgreSQL（数据库）
22. 代码托管使用自建 / 本地 GitLab；Preview 与 Prod 部署均采用 Docker Compose
23. Builder 与生成系统均以 Django 内置 auth 用户名密码方式登录，账号体系彼此独立
24. Agent 允许调用云端 LLM API，但 Builder 本体必须在内网部署；LLM 供应商通过配置切换
25. 候选版本 rc-N 按 PR 内部递增，不跨 PR 共享编号
26. Builder 本体部署采用 Docker Compose；MVP 不做 SaaS，长期可扩展
27. Fork 基座确认为 **OpenHands**（<https://github.com/All-Hands-AI/OpenHands>，MIT 协议）
28. Builder UI 采用"务实路线"：OpenHands 原有开发者向屏沿用其风格；TimberClaw 新增业务屏统一使用 Ant Design v5；两套风格共存，生成系统一律 AntD
29. **Owner 角色收紧为"工厂信息化负责人 / IT"**，有技术素养但不是开发者；业务使用者（End User）单列为第六个角色（§4.6）
30. **概念隔离原则**：工程域概念（commit / PR / rc / diff / migration / 堆栈 / 依赖名）**不得泄漏**到业务使用者 / Reviewer / 生成系统界面；Owner UI 必须按 §7.4 做术语映射
31. **Spec 双视图化**：业务场景视图（Owner 默认）+ 结构化专业视图（Platform Engineer / Admin）；同一份底层数据，两种呈现
32. **Dev 验证失败转人工 = Platform Engineer / Admin**，不是 Owner；Owner 只在需要改 spec 范围时被叫醒
33. **仪表盘是一等公民**：基于 Ant Design Charts，与列表 / 表单 / 详情并列；支持 4 类图表 + KPI 卡 + 明细 + 下钻；指标 / 维度来源于 spec，不做拖拽式设计器
34. **主数据以 CSV / Excel 单向导入维护**，业务结果支持 CSV 导出；MVP 不对接 ERP 接口，但模板底座必须内置导入 / 导出能力
35. **首期试点锁定离散制造**：工单（多工序）+ 派工 + 工序级报工 + 工序级质检（合格 / 让步 / 返工 / 报废）+ 基础仪表盘；流程制造与混合模式留到后续版本

## 15. Coding Agent 工作模式（V1.6 新增）

> 目的：让 Coding Agent 在没有人类逐步指令的情况下，也能从 `EXECUTION_TODO.md` 自主续作；同时对 Agent 设定明确的「停止条件」，避免在 PRD 边界外越权交付。

### 15.1 单一调度面（Single Source of Truth）

- **`timberclaw/docs/EXECUTION_TODO.md` 是当下唯一的调度面**。任何 Coding Agent 都必须从此文件挑选工单；不得依赖记忆 / 历史会话 / PR 评论作为下一步来源。
- **PRD（本文）是范围与约束的真相源**。当 EXECUTION_TODO 与 PRD 冲突时，以 PRD 为准；并由 Agent 在 PR 中提出对 EXECUTION_TODO 的更正。
- BACKLOG（`BACKLOG.md`）是工单语义来源；`ROADMAP_EXECUTION_PLAN.md` 是 Wave / 依赖图。三者由 EXECUTION_TODO 引用，不在 Agent 间「翻译」。

### 15.2 分支与 PR 协议

- **集成分支**：`cursor`。所有功能 PR 的 base 必须是 `cursor`，禁止直推 `main` / `cursor`。
- **工单分支命名**：`tc/<wave-id>/<short-desc>`（例：`tc/m0-wave-a/llm-gitlab-gateway`）。
- **单 PR 单工单**：每个 PR 对应 EXECUTION_TODO 上一张工单（W-x-NN）。如多张工单共享前置（如 Wave A 健康自检），允许串成 1 个收口 PR，但 PR 描述里必须列出覆盖的工单 ID。
- **批次（Batch）**：每张工单内允许多次 commit/批次；上限 5 批次。超限须拆分新工单。
- **PR 描述模板**：`Summary` / `Changes` / `Verification`（贴出本工单的 **Evidence** 命令与输出摘要） / `Merge target: cursor` / `Closes: <工单 ID>`。

### 15.3 工单状态机与 Pickup 协议

每张工单有四态：`pending` / `in_progress` / `blocked` / `done`。Coding Agent 续作流程：

1. **Pickup**：在 `EXECUTION_TODO.md` 内按下列优先级挑选下一张工单：
   1) 当前 Wave 中 `in_progress` 且未阻塞的工单
   2) 当前 Wave 中 `pending` 且依赖全部 `done` 的工单
   3) 若当前 Wave 全部 `done` 或仅剩 `blocked`，选择下一 Wave 中 `pending` 且依赖全部 `done` 的最早工单
   4) 若 1)~3) 都不满足，**停止**并报告（详见 §16.6）
2. **认领**：将该工单状态改为 `in_progress`，并在 commit 中以 `[tc-agent]` 标记；同一时间一张工单只能由一个 Agent 在改动。
3. **执行**：按工单的 **Range / DoD / Evidence** 推进；不得超出工单 Range；新发现需求**只追加新工单**，不得改写当前工单 Range。
4. **收口**：满足 DoD 后，将状态改为 `done`，把 Evidence 写入 PR 描述并合入 `cursor`。

### 15.4 每张工单的强制字段（EXECUTION_TODO 模板）

EXECUTION_TODO 中每张工单**必须**包含以下字段（缺字段视为工单未就绪，Agent 不得 Pickup）：

- **状态**：`pending` / `in_progress` / `blocked` / `done`
- **依赖**：上游工单 ID 列表（或 `无`）
- **关联 BACKLOG**：M-编号
- **范围（Range）**：本工单允许动到的代码 / 文档面（白名单）
- **Pickup 信号**：可机读的就绪条件（例如「`cursor` 上 W-A-04 = done」「`/api/health/llm/` 返回 `provider=mock`」）
- **DoD（Definition of Done）**：客观、可验收的条目（命令 + 期望输出 / UI 行为 / 文件存在 / CI 通过）
- **Evidence（必跑命令）**：Agent 在 PR 描述中必须粘贴的命令与摘要输出
- **不做**：明示越权项（防止把下一波次工作偷偷塞进来）

### 15.5 Wave 切换闸门

- 进入下一 Wave 前，**当前 Wave 的所有 W-x-NN 工单必须 `done` 或被显式 `blocked` 并登记原因**。
- Wave 切换需满足该 Wave 的 **DoD**（详见 EXECUTION_TODO 各 Wave 头部「Wave X DoD」）。
- 任何 Agent 不得跨 Wave 越级实现（例：Wave A 未收口前不准开 Wave B 工单）。

### 15.6 Agent 的停止条件（必须遵守）

Coding Agent 在以下情况**必须停止并向人类报告**：

1. EXECUTION_TODO 上没有可 Pickup 的工单（按 §16.3 规则）
2. 工单依赖被破坏（上游工单状态从 `done` 变 `blocked`）
3. PR CI（`.github/workflows/timberclaw-builder.yml` 等）连续失败 ≥ 2 次且原因不在工单 Range
4. 发现 PRD 与 EXECUTION_TODO 冲突
5. 工单 Range 不足以达成 DoD（必须先开新工单扩范围，而非偷扩范围）
6. 涉及 Prod 写入 / 高风险 migration / 删除上游目录 / 修改安全相关默认（凭据、CSRF、CORS）—— 一律停止并报告

### 15.7 默认 CI / 验证最小套件

每张 TimberClaw 后端 / 自检相关工单的 Evidence 至少包含：

- `docker compose build tc-backend && docker compose run --rm tc-backend python -m pytest`
- `bash scripts/tc_wave_a_check.sh`（要求 `docker compose up -d postgres tc-backend` 已就绪）

每张 TimberClaw 前端工单的 Evidence 至少包含：

- `cd frontend && npx eslint src/timberclaw --ext .ts,.tsx`
- 对应的 `vitest` / 截图（视工单而定，不强制全仓 vitest 通过）

## 16. 一句话定义

TimberClaw-code 是一个基于 **OpenHands** fork 改造的工业业务系统 Builder：它让工厂 IT（Owner）从业务场景出发，通过自然语言与 Agent 先生成并确认**spec 的业务场景视图**，再在 Dev 中基于 **React + Ant Design + Django + PostgreSQL** 的固定模板底座自动生成带有**仪表盘一等公民能力**和 **CSV 主数据导入 / 结果导出**的系统，在 Preview 中由业务代表（Reviewer）体验反馈，在人工把关下把最简可用、工业风格、不把工程概念暴露给业务使用者的系统发布到 Prod，最终由不懂代码的车间 / 排产 / 质检人员日常使用。
