# TimberClaw 持续开发总计划（M0 → M8）

> 目标：把 `BACKLOG.md` 中的 M0~M8 从“工单目录”推进为“可持续执行的工程计划”，用于日常迭代、并行分工、风险控制与验收闭环。
>
> 基线文档：`PRD.md`（V1.5）+ `BACKLOG.md`（V1.2）+ `CONVENTIONS.md`（V1.0）。

---

## 1. 执行原则（持续推进用）

1. **先打地基，再叠能力**：严格按 M0 → M8 的依赖顺序推进；仅允许在“依赖已满足”时并行。
2. **每个里程碑都要有可演示出口**：每完成一个里程碑，必须有“可运行、可验证、可回归”的演示脚本。
3. **最小化增量**：单次提交仅覆盖一个明确验收点，避免“跨里程碑混改”。
4. **概念隔离前置**：从 M1 开始，Owner/Reviewer/End User 可见文案统一纳入术语映射检查。
5. **高风险先判定后执行**：涉及 migration / 权限 / 模板底座 / 新依赖，一律先过风险门禁。

## 2. 里程碑节奏与波次规划

> 注：下面的“周次”是执行节奏建议，不是硬性日期承诺。实际按团队容量滚动调整。

### Wave A（基础设施，建议 2~4 周）
- **M0-01** Builder 项目脚手架（当前主线）
- **M0-02** 认证与账号
- **M0-03** LLM Gateway 抽象
- **M0-04** GitLab 接入
- **M0-05** 部署与自检

**出口定义（Wave A DoD）**
- `docker compose up` 可启动最小 Builder（OpenHands + Django + PG + 前端）。
- 可访问 OpenHands 原屏和 `/tc/*` 占位页。
- 可完成一次 GitLab 机器账号“分支→提交→MR”演练。
- 自检脚本可输出 PG / GitLab / LLM 三项连通状态。

### Wave B（需求到 spec，建议 2~3 周）
- **M1-01 ~ M1-06**：需求输入、初始 spec、对话修订、结构化编辑、确认闸门、双视图

**出口定义（Wave B DoD）**
- 同一份 spec 在“业务场景视图 + 结构化专业视图”可双向联动。
- 未确认 spec 无法进入生成流程。
- diff 可同时产出“业务摘要”和“结构化差异”。

### Wave C（模板与首次生成，建议 3~4 周）
- **M2-01 ~ M2-08**：前后端模板、部署模板、实现计划、首次代码生成、fixtures、仪表盘、主数据导入导出

**出口定义（Wave C DoD）**
- 基于确认 spec 可生成“工单 + 报工 + 质检 + 仪表盘 + 主数据导入”的可运行项目。
- Preview 能加载 fixtures。
- `external_import` 实体页面满足“只读 + 导入 + 导出”约束。

### Wave D（工程验证与协作，建议 2~3 周）
- **M3-01 ~ M3-05**：Dev 验证编排、自动修复、高风险识别、分支/MR 自动化、失败分派

**出口定义（Wave D DoD）**
- Dev 验证链路可结构化出错并分派给 PE/Admin。
- 低风险错误可自动修复（≤2 次）。
- Owner 只接收人话摘要，不接原始工程噪音。

### Wave E（候选版本与反馈闭环，建议 3~4 周）
- **M4-01 ~ M4-06** + **M5-01 ~ M5-05**

**出口定义（Wave E DoD）**
- Owner 可创建版本 N（rc-N 映射）并触发在线试用。
- Reviewer 可提交反馈，系统可按低/高风险分流到 patch 或 spec 修订。
- 术语映射扫描接入 Dev 验证。

### Wave F（发布与治理，建议 2~3 周）
- **M6-01 ~ M6-03** + **M7-01 ~ M7-06**

**出口定义（Wave F DoD）**
- Prod 发布具备人工确认与备注。
- 回滚支持“上一个生效版本”一键回退。
- 数据合同、写入边界、审计、备份机制上线。

### Wave G（业务验收与运营指标，建议 1~2 周）
- **M8-01 ~ M8-05**

**出口定义（Wave G DoD）**
- 离散制造首期五场景达到 PRD §13 指标基线。
- 成功标准看板可手工复核。
- PRD 对齐清单和 bug bash 结果可追踪。

## 3. 持续执行机制（每周循环）

### 3.1 周循环（建议）
1. **周计划**：从当前波次选择 3~5 个可交付子任务，声明依赖与验收。
2. **中期检查**：更新执行看板状态（pending/in_progress/blocked/done）。
3. **周验收**：跑“能跑的都跑”的验证命令，补齐风险记录。
4. **周复盘**：同步阻塞、决策、下周优先级。

### 3.2 提交策略
- 每个子任务一个 commit，便于回滚与审计。
- commit 强制含 `[tc-agent]`。
- 若触达上游目录，额外含 `[upstream-patch]` + 记录 `UPSTREAM_PATCHES.md`。

### 3.3 风险升级规则
- 连续 2 次自动修复失败：升级 Platform Engineer。
- 命中高风险改动：阻断自动推进，等待人工确认。
- PRD 冲突：先修文档，再写代码。
